GANG OF FOUR
-------------------------

The Gang of Four (GoF) refers to the four authors who wrote the influential book that introduced many of the classic software design patterns.

Design Patterns: Elements of Reusable Object-Oriented Software

The four authors are:

Erich Gamma
Richard Helm
Ralph Johnson
John Vlissides

Because there were four of them, they became known as the Gang of Four (GoF).

Why are they important?

Before their book, developers often solved common design problems repeatedly in different ways.

The GoF documented 23 proven design patterns that could be reused across object-oriented software.

Examples include:

Creational Patterns
------------------------
Factory Method
Abstract Factory
Builder
Prototype
Singleton


Structural Patterns
-------------------------
Adapter
Bridge
Composite
Decorator
Facade
Flyweight
Proxy


Behavioral Patterns
-----------------------------
Observer
Strategy
State
Command
Mediator
Memento
Iterator
Template Method
Chain of Responsibility
Visitor
Interpreter
Simple Analogy

Think of the GoF book like a cookbook.

Before the cookbook:

Every chef invented recipes from scratch.

After the cookbook:

Chefs recognized common recipes and reused them.

Similarly, the GoF didn't invent object-oriented programming—they identified and documented common solutions to recurring software design problems.

Key Idea (One-line Memory Aid)

"The Gang of Four are the four authors who documented the 23 classic object-oriented design patterns that are still widely used today."


BEHAVIORAL PATTERNS:
---------------------

1. Memento pattern

The Memento Pattern is used to save an object's state so it can be restored later, without exposing its internal details.

It's like taking a snapshot before making changes.

Analogy: Video Game Save Point
You're playing a game and reach a save point.
The game saves your current progress.
If you lose later, you can load the saved game and continue from that point.

Mapping:

Originator: The game (current progress)
Memento: Saved game file (snapshot)
Caretaker: Save/Load system that stores the snapshot
Key Idea (One-line Memory Aid)

"Take a snapshot now, so you can restore it later."

Real-world Examples
Undo/Redo in a text editor
Game save/load
Version history in a document editor

These all save a previous state so it can be restored later.

---------------------------------------------------------------------------------
2. State pattern

The State Pattern is used when an object's behavior changes based on its current state. Instead of using lots of if or switch statements, each state has its own behavior.

Analogy: Traffic Light

A traffic light behaves differently depending on its current state:

🟢 Green → Cars can go.
🟡 Yellow → Slow down.
🔴 Red → Stop.

The traffic light is the same object, but its behavior changes based on its state.

Mapping:

Context: Traffic Light
States: Green, Yellow, Red
Behavior: Go, Slow, Stop
Key Idea (One-line Memory Aid)

"Same object, different behavior depending on its current state."

Real-world Examples
Media Player: Playing, Paused, Stopped
ATM: No Card, Card Inserted, PIN Verified
Document: Draft, Review, Published

The object's behavior changes as its state changes.


-------------------------------------------------------------------------------------
3. Strategy pattern

The Strategy Pattern is used when there are multiple ways to perform the same task, and you want to choose the appropriate algorithm or behavior at runtime.

Instead of hardcoding the logic with if/switch statements, each strategy is implemented separately.

Analogy: Google Maps

You want to travel from A to B.

Google Maps lets you choose:

🚗 Drive
🚶 Walk
🚲 Bicycle
🚌 Public Transport

The destination is the same, but the strategy (route calculation) changes based on your choice.

Mapping:

Context: Google Maps
Strategies: Drive, Walk, Bicycle, Public Transport
Task: Find the best route
Key Idea (One-line Memory Aid)

"Same goal, different ways to achieve it—choose the strategy at runtime."

Real-world Examples
Payment methods: Credit Card, UPI, PayPal
Sorting algorithms: Quick Sort, Merge Sort, Bubble Sort
Compression: ZIP, RAR, 7Z

---------------------------------------------------------------------------
4. Iterator pattern

The Iterator Pattern is used to access each element in a collection one by one without knowing how the collection is stored internally.

Analogy: TV Remote

You don't need to know how the TV stores its channels.

You simply press:

▶️ Next Channel
◀️ Previous Channel

The remote lets you iterate through the channels one at a time.

Mapping:

Collection: TV Channels
Iterator: TV Remote
Elements: Individual Channels
Key Idea (One-line Memory Aid)

"Go through a collection one item at a time without knowing its internal structure."

Real-world Examples
foreach loop in C#
Iterating through a List<T>, Array, or Dictionary<TKey, TValue>
Database result sets
Easy C# Connection

When you write:

foreach (var item in list)
{
    Console.WriteLine(item);
}

You're using the Iterator Pattern. The foreach loop uses an iterator behind the scenes to move from one element to the next without you worrying about how the collection stores its data.

--------------------------------------------------------------------------
5. Command pattern

The Command Pattern is used to encapsulate a request or action as an object, so it can be passed around, queued, logged, or undone.

Instead of the caller performing the action directly, it creates a command object and asks it to execute.

Analogy: Restaurant
A customer places an order.
The waiter writes the order on a ticket (the command).
The chef prepares the food.

The customer doesn't talk directly to the chef. The waiter carries the command.

Mapping:

Client: Customer
Command: Order ticket
Invoker: Waiter
Receiver: Chef
Key Idea (One-line Memory Aid)

"Turn a request into an object so it can be executed later or by someone else."

Real-world Examples
Undo/Redo in editors
Menu or toolbar button clicks
Task queues and job schedulers
Remote controls (each button is a command)
Easy C# Connection

When you click a "Save" button in an application:

The button doesn't save the file itself.
It invokes a SaveCommand, which knows how to perform the save operation.

This keeps the UI separate from the business logic.

---------------------------------------------------------------------------------------------
6. Template pattern

The Template Method Pattern is used when multiple processes follow the same overall steps, but some of the individual steps are different.

The base class defines the overall workflow (the template), and derived classes customize specific steps.

Analogy: Making Tea vs Coffee

The overall process is the same:

Boil water
Add the main ingredient
Pour into a cup
Serve

The only difference is step 2:

🍵 Tea → Add tea leaves
☕ Coffee → Add coffee powder

The recipe (template) stays the same, but one step changes.

Mapping:

Template: Recipe
Concrete Implementations: Tea recipe, Coffee recipe
Variable Step: Add tea leaves or coffee powder
Key Idea (One-line Memory Aid)

"Same process, but some steps are customized by subclasses."

Real-world Examples
Generating different report types (PDF, Excel)
Data import process for different file formats (CSV, XML)
Game characters following the same turn sequence but performing different actions
---------------------------------------------------------------
7. Observer pattern

The Observer Pattern is used when one object needs to notify multiple other objects whenever something changes.

Instead of the other objects constantly checking for updates, they subscribe once and are automatically notified when a change happens.

Analogy: YouTube Subscription
A YouTube channel uploads a new video.
Everyone who has subscribed gets a notification automatically.
The channel doesn't need to know who all the subscribers are—it simply publishes the new video.

Mapping:

Subject (Publisher): YouTube Channel
Observers: Subscribers
State Change: New video uploaded
Notification: YouTube notifies all subscribers
Key Idea (One-line Memory Aid)

"One object changes → everyone interested is automatically notified."


---------------------------------------------------------------------
8. Mediator pattern

The Mediator Pattern is used when multiple objects need to communicate with each other, but instead of talking directly, they communicate through a 
central mediator.

This reduces dependencies between objects and keeps them loosely coupled.

Analogy: Air Traffic Control
Multiple airplanes need to take off and land.
Instead of pilots talking directly to every other pilot, all communication goes through the Air Traffic Control (ATC).
The ATC coordinates everything and tells each plane what to do.

Mapping:

Mediator: Air Traffic Control
Colleagues: Airplanes
Communication: Through the ATC, not directly with each other
Key Idea (One-line Memory Aid)

"Many objects communicate through one central coordinator instead of talking directly to each other."

------------------------------------------------------------------------------------------
9. Chain of Responsibility pattern

The Chain of Responsibility Pattern is used when multiple objects can handle a request, and the request is passed along a chain until one of them handles it.

Each object decides:

✅ "I can handle this."
❌ "I can't handle this, pass it to the next one."
Analogy: Customer Support Escalation

A customer has an issue.

Level 1 Support → Can't solve it? Pass to Level 2.
Level 2 Support → Can't solve it? Pass to Manager.
Manager → Resolves the issue.

The customer doesn't need to know who will solve it—the request moves through the chain until someone handles it.

Mapping:

Request: Customer issue
Handlers: Level 1 Support → Level 2 Support → Manager
Chain: The order in which the request is passed
Key Idea (One-line Memory Aid)

"Pass the request along until someone can handle it."

Real-world Examples
Customer support escalation
Expense approval (Manager → Director → CEO)
ASP.NET Core Middleware (each middleware handles or passes the request to the next)
Exception handling chains

10. Visitor Pattern

The Visitor Pattern lets you add a new operation to a group of existing classes without modifying those classes.

One-line memory trick:
"Keep the objects the same, bring in different visitors to perform different tasks."

🏛️ Analogy: Museum Visitors

Imagine a museum.

The museum has exhibits:
Dinosaur
Painting
Sculpture

These exhibits don't change.

Now different visitors come:

👨‍🏫 Guide → Explains history
🧹 Cleaner → Cleans the exhibit
📸 Photographer → Takes pictures
📝 Inspector → Checks for damage

The same exhibit receives different visitors, each performing a different operation.

The exhibit itself doesn't need to know how to clean or inspect—it simply accepts the visitor.

In Code

Instead of writing:

dinosaur.Clean();
dinosaur.Inspect();
dinosaur.TakePhoto();

You write:

dinosaur.Accept(cleanerVisitor);
dinosaur.Accept(inspectorVisitor);
dinosaur.Accept(photoVisitor);

Each visitor performs its own logic.

Why use Visitor?

Suppose you already have many classes:

Circle
Rectangle
Triangle

Now your manager says:

"Add exporting to PDF."

Instead of modifying all three classes, create:

PdfExportVisitor

Tomorrow:

ExcelExportVisitor

Next month:

AreaCalculationVisitor

The existing shape classes remain unchanged.

Visitor Pattern allows adding new operations to existing object structures without modifying the object classes by placing the operation inside separate Visitor classes.

11. Interpreter Pattern

The Interpreter Pattern is used to understand and evaluate a language or expression based on a set of grammar rules.

One-line memory trick:
"Given a language, build an interpreter that understands it."

🗣️ Analogy: Language Translator

Imagine you travel to Japan.

You say:

"Hello"

A translator interprets it as:

"こんにちは"

Similarly, if you type:

2 + 3 * 4

An interpreter understands the symbols (2, +, 3, *, 4) according to mathematical grammar and evaluates the result:

14

The interpreter reads, understands, and executes the expression.

Another Programming Analogy

Suppose your application accepts commands like:

MOVE FORWARD
TURN LEFT
MOVE FORWARD

Instead of writing lots of if-else statements, you create an interpreter that understands these commands and executes them one by one.

In Code

Input:

ADD 10 20

Interpreter:

Reads ADD
Understands it means addition
Computes:
30
Where is it used?
SQL query parsers
Regular expression engines
Mathematical expression evaluators
Programming language compilers/interpreters
Rule engines
Command/script processors
